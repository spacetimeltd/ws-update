# Components of an update data set:

1.) csv file containing update data in three parts

1.) updated 'new-item' records comprised of:

* new ID with corresponding 'Product-url', new image names and updated 'labels' values

* code appended with 'tmp'

* options codes appended with 'tmp'

* all other original data preserved

2.) original 'new-item' records made obsolete by update, modified thusly:

                                * code appended with 'old'

                                * options codes appended with 'old'

* path changed to new temporary container section named "TODELETE"
 ( This should remove the obsolete record from the location listed in the 'path' field. However, if the id appears in other locations it will have to be manually removed from the 'contents' of those pages.)

* 'labels' values updated

3.) leftover original 'new-item' records not intended for update in this set but containing 'labels' values that will be obsolete after the update (ie ids that are in this update set)

                                * 'labels' values updated to reflect the new ids

2.) list of ids from this update set that appear in 'contents' of ids that are not being updated in this set. These are products that show up on multiple pages. Since 'path' only contains one location and 'contents' data can't be  changed directly any other locations will have to have their 'contents' ids updated manually to complete the installation of each set.

3.) redirects csv containing redirects for the new ids

 

 

# Rollout Plan

Ideally, this is how it will work:

Step 1.  Import/upload update data set csv file
                This will add the new records with the updated ids, move the old records that are being replaced into a section named ‘TODELETE’, and update the ‘labels’ data for any records affected by the update but not updated in this set.

In order to avoid errors from duplicate ‘code’ values the new and old records’ code values (in both the ‘code’ field and ‘options’ field) are appended with ‘tmp’ and ‘old’ respectively. *** I’m not sure why I couldn’t just tack ‘old’ onto the old records and put the actually, unmodified (no ‘tmp’) code values in the new records. I think the yahoo system still recognized them, so after the catalog is published and the changes are made permanent then the ‘tmp’ ending can be removed from the new records. ***

Step 2. Manually update additional ‘contents’ data for products appearing in multiple locations not covered by the ‘path’ value.

Step 3. Delete old records now obsolete, should be as simple as deleting the ‘TODELETE’ section.

Step 4. Import redirects for the replaced ids.
